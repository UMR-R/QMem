Output ONLY valid JSON:

Core logic:
- The `episode` field is episodic memory for this conversation.
- `profile_updates`, `preference_updates`, `project_updates`, and `workflow_updates` are persistent-memory deltas.
- Be conservative: when in doubt, return fewer updates.

Rules:
- profile_updates: only fields that are new or clearly corrected.
- preference_updates: only newly expressed stable preferences or repeated usage-pattern signals.
- Put repeated "what the user often asks for" into `add_primary_task_types`, not profile.
- project_updates: only for user-owned ongoing projects, not reference papers/tools.
- workflow_updates: only for reusable recurring procedures.
- is_noise: true if the conversation has no durable memory value.

Episode rules:
- Keep the episode concise and factual.
- `related_project` should name the parent project only, or `""` if none.
- Do NOT include time fields in `episode`; the system preserves conversation date separately from source metadata at day granularity (`YYYY-MM-DD`).

Exclusion rules:
- Do NOT move objective identity facts into preferences.
- Do NOT move response preferences into profile.
- Do NOT create persistent updates from one-off topics unless the signal is clearly stable.
- Do NOT output extra keys, markdown, or explanations.
