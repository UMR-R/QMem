Core logic:
- The `episode` field is turn-level episodic memory for the new turn.
- `profile_updates`, `preference_updates`, `project_updates`, and `workflow_updates` are persistent-memory deltas.
- Be conservative: when in doubt, return fewer persistent updates while still preserving the turn as an episode if it has durable value.

Language policy:
- The rules are in English for precision.
- Natural-language output should follow TARGET DISPLAY LANGUAGE if supplied, otherwise the dominant language of the new turn.
- If the output language is Chinese but an English term is clearer or is a proper noun, keep that term in English.
- Preserve paper/model/dataset/tool names, code names, and technical terms in their original form.

Temporal rules:
- The user prompt may include timestamp, event order, current memory timestamps, or previous episode order. Use those signals to interpret "current", "previous", "before", "after", "latest", "old", and "now".
- If a new statement corrects or supersedes older memory, output only the update needed to reflect the newer state.
- Do NOT output time fields. The system stores timestamps from source metadata.

Persistent update rules:
- profile_updates: only stable identity, role, background, language mode, or durable communication context that is new or clearly corrected.
- preference_updates: only stable response, interaction, format, terminology, language, or style preferences.
- Put repeated broad activity modes into `add_primary_task_types`; do not put concrete objects, products, papers, datasets, tools, locations, or one-off topics there.
- project_updates: only user-owned ongoing projects, not reference papers/tools.
- workflow_updates: only reusable recurring procedures. If a review/check step has its own trigger and can be reused independently, represent it as a separate workflow update instead of hiding it inside a larger workflow.
- is_noise: true only if the turn has no durable memory value.

Episode rules:
- Keep the episode concise and factual, describing only the new turn.
- Use current memory only as context; do not blend unrelated older facts into the new episode.
- `related_project` should name the parent project only, or `""` if none.
- Do NOT include time fields in `episode`.

Evidence boundary rules:
- Do NOT move objective identity facts into preferences.
- Do NOT move response preferences into profile.
- Do NOT create persistent updates from one-off topics unless the signal is clearly stable.
- Assistant-only suggestions are not user decisions. Put them in summary or next_actions/open issues unless the user explicitly accepts them.
- Do not copy assistant-generated labels, positioning, or proposed strategies into persistent fields as confirmed user facts.
- Do NOT output extra keys, markdown, or explanations.
