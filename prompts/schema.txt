# Portable Personal Memory Layer - Architecture Definition v1.3

## Core Layers

The system separates memory into evidence and persistent knowledge.

### L0 Raw Conversations

Raw conversations are the canonical source evidence captured from chat UIs or platform exports.

- `RawChatSession` groups turns from one conversation/session and usually maps to one JSON file.
- `RawChatTurn` is the minimum raw evidence unit.
- A turn contains one user side, one assistant side when available, and one shared timestamp.
- If one side is missing, preserve the turn with a status marker instead of silently dropping it.

### L1 Platform Memory Signals

Platform-provided memory or configuration signals, such as:

- saved memory
- custom instructions
- agent/GPT config
- platform skills
- profile/preferences exposed by the platform

L1 is not the project's canonical persistent store. It is an input signal.

### L2 Episodic Memory

Episode memory is turn-level structured evidence distilled from L0.

- One episode describes exactly one raw turn.
- It preserves what happened in that turn: task, context, decisions, open issues, and relation flags.
- It keeps source references back to raw session/turn ids.
- Timestamps are maintained from source metadata by the system, not invented by the model.
- Episodes can connect to neighboring turns and to other directly related episodes.

### L2 Persistent Memory

Persistent memory is stable knowledge distilled from episodes and L1 signals.

- It represents user profile, preferences, projects, workflows, daily notes, and skills.
- It must keep evidence links back to episodes or raw turns.
- It must keep confidence and export priority.
- It is the canonical memory store for export, display, and index generation.

### L3 Policy / Governance

L3 defines how memory is validated, linked, merged, split, exported, and indexed.

- It governs cross-episode connection validation.
- It prevents accidental mega-groups caused by weak transitive links.
- It decides when a memory should remain episodic versus become persistent.
- It controls derived external indexes such as ChromaDB.

## Raw Turn Schema

```json
{
  "turn_id": "conversation_id:turn:0",
  "session_id": "conversation_id",
  "timestamp": "2026-04-24T12:00:00Z",
  "user_text": "",
  "assistant_text": "",
  "status": "complete | missing_user | missing_assistant"
}
```

## Episode Schema

```json
{
  "episode_id": "ep_0001",
  "session_id": "conversation_id",
  "turn_refs": ["conversation_id:turn:0"],
  "title": "episode title",
  "topics_covered": ["topic"],
  "summary": "turn-level summary",
  "key_decisions": ["decision explicitly made in this turn"],
  "open_issues": ["unresolved issue"],
  "relates_to_profile": true,
  "relates_to_preferences": false,
  "relates_to_projects": ["parent project"],
  "relates_to_workflows": ["workflow"],
  "time_range_start": "2026-04-24",
  "time_range_end": "2026-04-24",
  "created_at": "2026-04-24",
  "updated_at": "2026-04-24"
}
```

## Episode Time Rules

- Episode time comes from the raw turn timestamp.
- If only one timestamp exists, use it for start/end and created/updated.
- Use event order when timestamps are missing.
- The model should use time/order to understand "current", "previous", "before", "after", "latest", and "old", but the system writes the actual time fields.

## Persistent Node Schema

```json
{
  "type": "preference | profile | workflow | topic | daily_note | skill",
  "key": "global_snake_case_key",
  "description": "stable memory description",
  "episode_refs": ["ep_0001", "ep_0002"],
  "turn_refs": ["conversation_id:turn:0"],
  "confidence": "low | medium | high",
  "export_priority": "low | medium | high",
  "platform": ["source platform"],
  "updated_at": "2026-04-24"
}
```

## Persistent Type Boundaries

| type | Meaning |
|---|---|
| `profile` | Stable identity, role, background, language mode, and long-term domain context |
| `preference` | Stable response style, terminology, formatting, language, revision, and interaction preferences |
| `topic` | Long-running project theme, research/product direction, or durable interest |
| `workflow` | Reusable procedure with trigger and ordered steps |
| `daily_note` | Reusable non-project daily context, personal choice context, taste, lifestyle, or small useful facts |
| `skill` | Saved or recommended Skill asset that the user explicitly wants to keep or reuse |

## Confidence Rules

| supporting evidence | confidence |
|---|---|
| one weak/one-off signal | `low` |
| two to three consistent signals | `medium` |
| four or more consistent signals, or explicit platform saved memory | `high` |

Confidence should match evidence strength and should not be inflated by assistant-only suggestions.

## External Index Rule

External databases such as ChromaDB are derived indexes.

- They are built from canonical raw, episode, and persistent stores.
- They are not the authority for memory content.
- They may use different metadata fields for retrieval or benchmark settings.
- They can be rebuilt from canonical storage at any time.
